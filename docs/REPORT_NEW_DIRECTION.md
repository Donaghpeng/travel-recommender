## 一、现状诊断：为什么 OpenClaw 反复犯错？

读完全部代码后，我发现问题的根源**不是 AI 能力不够，而是三个结构性缺陷**让任何 AI 助手都容易犯错：

### 🔴 缺陷 1：前端没有「安全网」

```js
// app.js — 1755 行，0 行测试
// 现状：全部是全局变量 + DOM 直接操作
var selectedIdx = [];
var currentResults = [];
```

- `app.js` 80KB 无任何单元测试
- OpenClaw 改完 A 功能 → 不知道 B 功能是否被破坏
- 没有「改完跑一下测试就能知道对不对」的机制
- 所以同样的 POI 展示 bug 反复出现 → 因为没人告诉它又坏了

### 🔴 缺陷 2：代码债在不断累积

```python
# app.py 361-477 行：大量已注释/无效代码
# ─── 美团异步任务池 ─────────
# ─── 美团城市级缓存 ────────
# 启动时自动预加载的热门城市...

def _kill_process_tree(proc):  # 美团残留，但已无调用
    ...
```

- `scripts/` 目录下 30+ 个一次性调试脚本（`fix_*.py`, `check_*.py`, `find_*.py`）
- `app.py` 有 100+ 行死代码（美团的遗迹）
- 每次 AI 读代码都会被这些「看似有用实则已废」的代码干扰判断

### 🔴 缺陷 3：缺少「功能清单 + 冒烟测试」

没有一个文件明确说：

> 「联游功能是这样的：API 端点 X → 前端函数 Y → 渲染方式 Z」

AI 只能从代码里猜，猜错就改坏。

## 二、核心问题：三个瓶颈

| 问题 | 表象 | 根源 |
|:----|:----|:-----|
| 🎯 **推荐同质化** | 每次推荐都差不多那几张面孔 | **58 个目的地太少**（计划扩到 300-500） |
| 🐛 **功能更新丢失** | 联游功能丢了、POI 展示反复出 bug | **无测试 + 无状态管理** |
| 📱 **演示不够惊艳** | 首屏空数据、交互不够顺滑 | **异步 Enrich 模式的副作用** |

## 三、新方向：「稳定基座 + 数据飞轮」

放弃「堆功能」的思路，转向 **先求稳、再求大、最后求美** 的三阶段路线：

```
当前痛点                       新方向
┌─────────────┐         ┌─────────────────────┐
│ ❌ 功能堆砌   │  ──→   │ ✅ 测试兜底 + 稳定迭代│
│ ❌ 58 个目的地│  ──→   │ ✅ 300+ 目的地数据飞轮│
│ ❌ 反复修bug  │  ──→   │ ✅ 架构清理 + 自动化  │
└─────────────┘         └─────────────────────┘
```

### 核心原则

1. **改之前先有测试** — 任何 AI 助手改代码前，先跑测试知道「原来是对的」
2. **功能清单驱动** — 每个功能有明确的规格说明，AI 照着做
3. **数据先行** — 推荐系统的灵魂是数据，不是前端特效
4. **渐进增强** — 不动稳定部分，新功能以「插件」方式追加

## 四、Phase 1：基座加固（1-2 周）

> **目标：让 AI 不再「越改越坏」，让演示不再「首屏空白」**

### Step 1.1：建立功能清单 ✅

创建 `FUNCTIONAL_MAP.md`，明确记录每个功能：

| 功能 | 前端入口 | 后端 API | 状态 |
|:----|:---------|:---------|:-----|
| 目的地推荐 | `doSearch()` → `renderResults()` | `GET /api/recommend` | ✅ 稳定 |
| POI 景点搜索 | `doPOISearch()` → `renderPOIResults()` | `GET /api/poi-search` | ⚠️ 展示问题 |
| POI→推荐打通 | POI 卡片底部按钮 → 切换 Tab | 无需新 API | ❌ 丢失 |
| 多城市联游 | `tab-multi` | 待确认 | ❌ 丢失 |
| 天气预警 | ⚠️ 工具栏图标 | `GET /api/weather/warning` | ✅ |
| 侧滑抽屉 | `openDrawer()` → `renderDrawer()` | 纯前端 | ⚠️ 内容填充 |
| ... | ... | ... | ... |

### Step 1.2：前端「安全网」🛡️

不用大改架构，加 3 个轻量级防护：

```javascript
// 1. function-registry.js — 登记所有关键函数
window.__FUNCTIONS__ = {
  doSearch: { deps: ['renderResults', 'postRender'] },
  renderResults: { deps: ['bgGrad', 'sc'] },
  doPOISearch: { deps: ['renderPOIResults', 'injectCityWeatherToPOI'] },
  // ...
};

// 2. 在 dev 模式下校验：改完后这些函数还存在吗？
```

```python
# 3. smoke_test.py — 3 秒跑完的冒烟测试
# 启动服务器 → 调每个 API → 验证返回结构
# 跑完只要 3 秒，每次改完跑一下
```

**为什么这能解决「反复犯同样错误」？**
- OpenClaw/Hermes 改代码后，`smoke_test.py` 立刻报错："POI 搜索返回格式不对"
- AI 看到测试失败就知道自己改坏了

### Step 1.3：代码清理 🧹

| 清理项 | 文件 | 操作 |
|:-------|:-----|:-----|
| 美团死代码 | `app.py` L361-477 | 删除无效函数 |
| 调试脚本 | `scripts/` | 移入 `scripts/archive/` |
| 无效路由 | `app.py` | 检查并清理无调用的端点 |
| 空函数/死注释 | 全局 | 清理 |

### Step 1.4：首屏体验优化 ⚡

当前问题：首次请求无 AI 描述、无天气详情 → 需刷新

**方案 A（推荐）**：启动时预热
- `app.py` 已有 `_startup_warm_search_cache()`  
- 扩展它：启动时预计算热门出发地 + 默认偏好的完整结果（含 AI + 天气）
- 用户首次打开 → 直接命中缓存 → 全量数据

**方案 B（备选）**：前端骨架屏升级
- 当前已有 skeleton-card CSS
- 后端返回后逐步填充（先评分 → 再 AI → 再天气）
- 用动画过渡，不闪烁

### Step 1.5：修复联游 + POI 打通 🔧

这部分是「丢失的功能」，需要：
1. 确认联游 API 和前端 Tab 的对接
2. 修复 POI 按钮 → 跳转推荐的逻辑
3. 加上测试，防止再次丢失

## 五、Phase 2：数据飞轮（2-4 周）

> **目标：58 → 200+ 目的地，让推荐每次都有新鲜感**

### Step 2.1：执行目的地扩增 🗺️

沿用 `docs/destination-expansion-plan.md` 的方案：

```
高德 POI API ──→ fetch_amap_poi.py ──→ 200+ 目的地
                      ↓
DeepSeek API ──→ generate_descriptions.py ──→ 丰富描述
                      ↓
                  加载到系统
```

**先跑 5 省测试**（江浙沪 + 广东 + 四川），验证质量后再全量。

### Step 2.2：推荐多样性微调 🔄

当前 MMR 只防同类目的地连续出现。增加三层候选池：

```
核心池（80%）：高评分 → 稳定推荐
探索池（15%）：中等评分 → 偶尔出现新面孔
新鲜池（5%）：新增目的地 → 给曝光机会
```

### Step 2.3：用户反馈闭环 🔁

- `/api/feedback` 已有，但数据未被利用
- 建立：用户点击「喜欢」→ 记录偏好 → 下次推荐权重微调
- 这是面试时可以讲的技术亮点：「系统从用户行为中学习」

面试官会问的：
- 你为什么选这个架构？
- 评分引擎的 5 个维度怎么权衡？

- PR 指标（推荐结果多样性、覆盖率等）

## 七、技术架构建议

### 当前架构（需要保留的优点）

```
同步评分 → 毫秒级返回 → 后台 Enrich
         → 缓存 LRU + TTL
         → 优雅降级（所有外部 API 都有 fallback）
```

这些都是好的设计，面试时要能讲清楚。

### 建议增补

| 增补项 | 为什么 |
|:-------|:-------|
| **统一的前端状态管理** | 用简单的发布订阅模式替代全局变量 |
| **API 契约测试** | 确保前端/后端对接稳定 |
| **功能开关（Feature Flag）** | 新功能默认关闭，验证后再开启 |
| **数据版本控制** | 目的地数据有版本号，升级不破坏旧缓存 |

## 八、如果让 Hermes 来做 vs OpenClaw

| 维度 | OpenClaw 做法 | Hermes 做法 |
|:----|:-------------|:------------|
| **每次修改前** | 直接改代码 | 先跑测试，确认基线 |
| **改完后** | 你必须手动检查 | 自动跑冒烟测试，不过关回滚 |
| **新功能** | 直接嵌入现有代码 | 以「插件」方式追加，不动核心 |
| **犯错成本** | 高（丢失功能你不知道） | 低（测试会告诉你） |
| **长期** | 代码越来越乱 | 每次修改都在改善结构 |

**关键差异**：  
我先把**功能清单**和**冒烟测试**建好 → 在安全网里改代码  
OpenClaw 直接改 → 没有安全网 → 改坏了你也不知道
